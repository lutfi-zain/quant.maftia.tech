## Context

The `quant-proxy` Cloudflare Worker sits between users and the quant.maftia.tech application. It receives all traffic for `quant.membran.app` via a `custom_domain` route and proxies requests to the origin server at `43.133.148.181:sslip.io:8911` (Vite dev server, which proxies `/api` and `/ws` to the API Gateway on `:8910`).

The Worker applies a header allowlist filter for security, then uses standard `fetch()` to proxy the request to the origin. This works correctly for HTTP traffic.

However, WebSocket upgrades are broken because:

1. **Header filtering strips WebSocket headers** — `Upgrade`, `Sec-WebSocket-Key`, `Sec-WebSocket-Version` are not in the `safeHeaders` allowlist, so they are silently removed before forwarding.

2. **`fetch()` cannot perform WebSocket upgrades** — Cloudflare Workers' `fetch()` API sends a regular HTTP request, not a WebSocket handshake. The origin receives a plain GET with no WebSocket semantics, and the Hono/Bun server returns `400 "WebSocket upgrade failed"`.

Diagnosis confirmed: direct WebSocket connections to any origin port (`:8910`, `:8911`, `:5173`) work perfectly. Only connections routed through the Worker fail.

Additionally, Vite's HMR client auto-detects the WebSocket URL from the dev server config. When accessed through a reverse proxy domain (`quant.membran.app`), it generates `ws://localhost:8911` (the server's address) instead of `wss://quant.membran.app`. This must be explicitly configured.

## Goals / Non-Goals

**Goals:**

- All WebSocket connections through `quant.membran.app` establish successfully: `/ws/live` (app real-time data), Vite HMR, and any browser extension WebSockets
- Bidirectional message flow works for subscribed topics (analytics updates, circuit breaker broadcasts)
- Header allowlist continues to filter unnecessary or dangerous headers
- Zero changes to the API Gateway (`src/api/`), frontend websocket hook (`useTerminalWebSocket.ts`), or quant engines

**Non-Goals:**

- No changes to WebSocket topic structure, broadcast logic, or message format on the API Gateway
- No changes to port assignments (`:8910` API, `:8911` Vite, `:5173` socat)
- No changes to the existing Cloudflare dashboard configuration
- No new deployment infrastructure (Keepalived, load balancers, etc.)

## Decisions

### Decision 1: Use `WebSocketPair` API instead of `fetch()` for WebSocket upgrades

**Chosen:** Use Cloudflare Workers' `WebSocketPair` API to create a bidirectional WebSocket tunnel between the client and the origin server.

**Rationale:** `WebSocketPair` is the officially supported mechanism for proxying WebSocket connections in Cloudflare Workers. It creates a pair of connected WebSocket instances: one returned to the client (as a 101 Switching Protocols response) and one used internally to connect to the backend.

**Alternatives considered:**

| Approach | Pros | Cons |
|---|---|---|
| `WebSocketPair` (chosen) | Standard CF Workers pattern; full bidirectional message relay; works with existing Bun/Hono WS handler | Slightly more code than the broken `fetch()` approach; requires manual `accept()` and event piping |
| Direct `new WebSocket(url)` | Simpler one-liner | Can't be returned as the response WebSocket directly; requires `WebSocketPair` for the client side anyway |
| Remove the Worker, let Cloudflare proxy directly | No Worker code change needed; Cloudflare's built-in WebSocket proxy works natively | Loses header filtering; requires Cloudflare dashboard config change; IP flexibility reduced |

### Decision 2: Route WebSocket through Vite dev server (`:8911`) not directly to API Gateway (`:8910`)

**Chosen:** Keep the existing routing architecture — Worker → Vite (`:8911`) → API Gateway (`:8910`) for WebSocket, matching the HTTP path.

**Rationale:** Vite's proxy rule for `/ws` already handles the WebSocket upgrade correctly (verified with direct connections). Keeping the same path for HTTP and WS ensures consistency. The alternative (routing WS directly to `:8910`) would require path differentiation and create asymmetry.

### Decision 3: Bypass header allowlist for WebSocket upgrade requests

**Chosen:** When forwarding a WebSocket upgrade, include all original WebSocket headers (`Upgrade`, `Sec-WebSocket-Key`, `Sec-WebSocket-Version`, `Sec-WebSocket-Extensions`, `Connection`) in addition to the standard safe headers. These headers are mandatory for the WebSocket handshake and carry no security risk (they are generated by the WebSocket API, not user-controlled).

**Rationale:** Without these headers, the origin cannot complete the WebSocket handshake. The headers are produced by the browser's WebSocket implementation and are not user-injectable in a meaningful way.

### Decision 4: Configure Vite HMR explicitly in `vite.config.ts`

**Chosen:** Add `server.hmr` configuration to point to `quant.membran.app:443` with `wss` protocol.

**Rationale:** Without explicit configuration, Vite's auto-detection generates `ws://localhost:8911` in the HMR client, which the browser can't reach (localhost from the browser is the user's machine, not the server). Explicit `server.hmr` injects the correct URL into the client bundle.

**Alternatives considered:**

| Approach | Pros | Cons |
|---|---|---|
| Explicit `server.hmr` (chosen) | Simple, predictable, works with any reverse proxy | Hardcodes the domain name |
| `server.hmr.clientPort` | More portable across hostnames | Doesn't change protocol (http→ws vs https→wss) |
| `allowedHosts` + magic | No config change needed | Doesn't fix the core URL issue; Vite still generates wrong URL |

### Decision 5: Connect to origin using backend WebSocket URL matching the request path

**Chosen:** The Worker will create a `WebSocket` connection to `ws://43.133.148.181.sslip.io:8911{path}` preserving the original request path (e.g., `/ws/live`, `/` for Vite HMR).

**Rationale:** The Vite dev server needs the exact path to route correctly. Vite's proxy rule for `/ws` matches on the path `/ws/live`, and the HMR WebSocket connects to the root path with query parameters.

## Risks / Trade-offs

| Risk | Likelihood | Mitigation |
|---|---|---|
| **WebSocketPair message backpressure** — if the origin sends data faster than the Worker can forward to the client, memory grows unbounded | Low | Cloudflare Workers runtime manages backpressure internally for `WebSocketPair`; the thin relay introduces negligible latency |
| **Worker CPU/execution time** — proxying every message through the Worker adds overhead vs direct Cloudflare WebSocket proxy | Low | The Worker is already in the request path for HTTP; WebSocket relay is lightweight string forwarding |
| **Connection cleanup on Worker eviction** — if the Worker isolate is evicted (rare), WebSocket connections drop | Medium | The API Gateway's heartbeat interval (30s ping, 60s stale cleanup) handles this gracefully; the frontend `useTerminalWebSocket` hook has exponential backoff reconnection |
| **sslip.io DNS resolution changes** — if the server IP changes and sslip.io record is updated, WebSocket connections to old IP hang | Low | Same risk exists for HTTP traffic; the Worker uses the same `sslip.io` hostname for WS; reconnection handles stale connections |
| **Vite HMR path collision** — if the Worker intercepts ALL WebSocket upgrades, Vite HMR (`/`) and app WS (`/ws/live`) both go through the same handler | None | This is correct; the Worker path-preserving approach means each WebSocket connects to the right origin path |
